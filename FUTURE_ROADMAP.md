### v2.13.0 — 🌙 深色模式（H1）+ 🔐 App Check（C1）（2026-05-18）

> **設計系統紅利收割 + 安全強化版**。一發兩個建議組合：H1 把 v2.11 預留好的 `[data-mode="dark"]` token 接管完成；C1 接續 v2.10 通知基建從「告警」進化到「事前阻擋」。

#### 主要新增

**🌙 深色模式（H1）**
- 完整接管 `[data-mode="dark"]` CSS variables（v2.11 設計階段已預埋）
- `useTheme` hook：`light` / `dark` / `system` 三段切換
  - localStorage 即時 + Firestore `settings/user.theme` 跨裝置同步
  - `system` 模式自動跟 `prefers-color-scheme` 切換
  - 雲端值 win（讓多裝置使用者體驗一致）
- `bootstrapTheme()` 在 React 渲染前同步套用，**避免亮 / 暗閃爍（FOUC）**
- 三個 UI 切換點：
  - Header avatar 旁的 Sun/Moon/Monitor 快速按鈕（一鍵 cycle）
  - User dropdown 的 3-option segmented control（精確選擇 light/dark/system）
  - PWA `theme-color` meta 同步切換（手機狀態列也跟著變）
- 智慧 CSS override：`bg-white` / `text-black` / `bg-gray-*` 等 Tailwind 灰階在 dark mode 自動 redirect 到 token，省得 sweep 27 個檔案
- 0.2s 平滑色彩 transition，切換不刺眼
- 全套涵蓋：`.bg-paper` noise pattern、`.bg-grid`、scrollbar、`.b-dash`、`.tape`、`.bg-lined` 全部都有 dark 變體

**🔐 App Check（C1）— Phase 1 fail-open**
- 前端 `initializeAppCheck` 整合 reCAPTCHA v3 provider
- Site key 走 `VITE_APP_CHECK_SITE_KEY` env，**未設定時跳過**（不擋老師使用）
- Dev 環境自動啟用 `FIREBASE_APPCHECK_DEBUG_TOKEN`，本地測試不被 reCAPTCHA 卡
- `cloudFunctionsApi.js` 每個 fetch 自動帶 `X-Firebase-AppCheck` header（best-effort）
- 後端 middleware 三段模式 `APP_CHECK_MODE: 'off' | 'observe' | 'enforce'`：
  - `off`（預設）：完全跳過，不影響任何流量
  - `observe`：驗證但 invalid 只 log warning 不擋（觀察期看誤殺率）
  - `enforce`：嚴格 enforce，invalid 直接 403
- 設計確保「Phase 1 部署後零行為改變」，等使用者設好 site key + 觀察 1-2 天後再切 enforce

#### 啟用 SOP（App Check）

**步驟 1 — Firebase Console 註冊**：
1. https://console.firebase.google.com/project/comments-67079/appcheck
2. Build → App Check → Register Web App
3. 選 reCAPTCHA v3（免費，門檻最低）
4. 拿到 Site Key（`6L...` 開頭的字串）

**步驟 2 — 把 Site Key 設成環境變數**：
- 本地：`.env.local` 加 `VITE_APP_CHECK_SITE_KEY=6L...`
- GitHub Actions：repo Settings → Secrets → 加 `VITE_APP_CHECK_SITE_KEY`，並在 build workflow 把它寫進 env

**步驟 3 — 觀察期切換**（1-2 天後）：
1. 改 `functions/src/middleware/auth.ts` 內 `APP_CHECK_MODE` 從 `'off'` → `'observe'`
2. 部署：`firebase deploy --only functions:api --account cagooo@gmail.com --project comments-67079`
3. 觀察 Firebase Console → App Check → Metrics 1-2 天
4. 確認所有合法流量都有 token 後，切到 `'enforce'`

### 🅗 設計系統延伸（從新建的 token / atoms 出發）

#### H1. 深色模式（Dark Mode）✅ **已完成 v2.13.0**
**實際工時**：~3 小時 ｜**價值**：超高

設計檔的 `HANDOFF.md` 已經把所有顏色的 dark 變體都列出來了（`--ink` 在 dark 是 `#FCEEC9`，`--paper` 是 `#1E1A14` 等），但 `tokens.css` 還沒接上。**這代表 50% 已經完成，剩下的只是「接管 data-theme 屬性」**。

```
實作步驟：
├── tokens.css 加 [data-theme="dark"] { ... } block（複製 HANDOFF.md 的對照表）
├── settingsService 加欄位：theme: "light" | "dark" | "system"
├── 在 App.jsx mount 時讀偏好 → document.documentElement.dataset.theme
├── Header 加切換按鈕（用 Sun / Moon Icon，已在 lucide-react）
└── prefers-color-scheme media query 監聽，"system" 模式自動跟隨
```

**為什麼最高 ROI**：
- token 已抽乾淨，**完全不需要動元件程式碼**
- 老師夜間備課常見場景
- 展示「設計系統正確抽象」的最佳例子

#### H2. 設計系統 Storybook 化 ⭐⭐⭐
**預估**：1-2 天 ｜**價值**：高（設計債清零）

- Btn：5 種 tone × 3 種 size × disabled/loading 狀態
- Chip：5 色 token + selected/unselected
- Card：3 種 padding × 有無 sticker × 有無紙膠帶

**好處**：

### 🅓 後端能力延伸

#### D1. 批次任務佇列化（Cloud Tasks）⭐⭐⭐
**預估**：1 週 ｜**價值**：高

目前 `processBatchJob` 是「fire-and-forget」，最大缺點：
- 單一 Cloud Function 最多 9 分鐘，100 位學生 × 5 秒 = 500 秒會被砍
- 失敗無法自動 retry

改用 Cloud Tasks 把每位學生 dispatch 成獨立 task：
- `handleCreateBatch` 收到 → 建立 `batchJobs/{jobId}`
- 對每位學生呼叫 `cloudTasks.createTask({ url: '/internal/generateOne', payload: {...} })`
- 每個 task 獨立執行，失敗自動 retry
- 前端輪詢 `batchJobs/{jobId}` 看進度

**好處**：可靠性大躍進、可視化進度、不再被 timeout 砍。

#### D2. Streaming 評語生成 ⭐⭐⭐⭐ — **UX 大躍進**
**預估**：1-2 天 ｜**價值**：超高

目前老師按「生成」要等 3-8 秒看到結果（黑屏感）。改 Server-Sent Events：

```
前端: const eventSource = new EventSource('/generate?stream=1');
       eventSource.onmessage = (e) => appendToComment(e.data);

後端: Gemini API 已支援 streamGenerateContent
       Cloud Function 用 ReadableStream 把 chunk 即時推給前端
```

效果：字一個一個浮現（像 ChatGPT），等待感消失。**搭配 H4 動效，是 v2.11 設計系統最佳搭擋**。

#### D3. 快取重複請求 ⭐⭐
**預估**：3 小時 ｜**價值**：中（看流量決定）

新增 `cache/{hash}` collection，TTL 7 天：
```
hash = md5(traits + styles + tone + wordCount)
generate 前先查，命中就直接回（標註「（快取結果）」）
```

#### D4. 多 LLM 後備（Failover）⭐⭐
**預估**：1 天 ｜**價值**：中

Gemini 429 / 503 時自動 fallback 到 Claude 或 OpenAI：
```typescript
async function generateWithFallback(prompt) {
  try { return await gemini.generate(prompt); }
  catch (err) {
    if (isQuotaErr(err) || isUnavailable(err)) {
      notifyAdmin('🔄 Gemini fallback 觸發');
      return await claude.generate(prompt);
    }
    throw err;
  }
}
```

#### D5. Cost Tracking（精確到 token）⭐⭐
**預估**：4 小時 ｜**價值**：中

Gemini API 回應已含 `usageMetadata.totalTokenCount`，寫進 `users/{uid}/usage/{date}`：
```
├── promptTokens
├── completionTokens
├── totalTokens
└── estimatedCostUsd  # Gemini 2.5 Flash 單價計算
```
管理員月底彙整 → LINE 推「本月成本 NT$XX」。

#### D6. Cloud Functions Region 切到 asia-east1 ⭐⭐
**預估**：1 小時 ｜**價值**：低-中（latency 微優化）

如果目前還在 us-central1，切到 asia-east1（彰化）可省 150-200 ms RTT。注意要重新部署所有 secrets 並更新前端 URL。

- Google 登入 → 申請 → 自動審核 → 看到主畫面
- 新增學生 → 選成語 → 生成評語 → 編輯 → 儲存
- 批次匯入 Excel → 批次生成 → 匯出

#### TD4. 虛擬滾動 ⭐⭐
**預估**：2 小時

班級 > 50 人時 StudentRow 卡頓。用 `react-window` 或 `tanstack-virtual`。

#### TD5. 錯誤邊界強化 ⭐⭐
**預估**：1 小時

分區 Error Boundary：Header / 學生列表 / Modal 各一個，避免單一 modal 炸掉整頁。

#### TD6. console.log 清理 ⭐
**預估**：30 分鐘

Vite plugin `vite-plugin-remove-console` 自動 prod build 刪除。

#### TD7. Bundle 分析 / 拆 chunk ⭐
**預估**：1 小時

`vite-plugin-visualizer` 分析，找出大頭：
- `firebase/firestore` 約 200 KB
- `xlsx` 約 600 KB（最大！只在 import 時才需要）
- `jspdf` + `html2canvas` 約 400 KB（只在列印才需要）

把 xlsx / pdf 改 dynamic import，初次載入可砍 1 MB。

#### TD8. CI / Lighthouse 自動跑分 ⭐⭐
**預估**：2 小時

GitHub Actions 每次 PR 跑 Lighthouse CI，分數低於 80 自動 fail。

### 🅘 PWA / 行動體驗

#### J1. PWA 推播通知（Web Push）⭐⭐⭐
**預估**：1 天 ｜**價值**：高

老師裝 PWA 後可收推播（評語批次完成、配額預警），比 LINE 通知門檻更低（不用綁定）。需要 VAPID key + service worker。

#### J2. Capacitor 打包成 App ⭐⭐
**預估**：6-8 小時 ｜**價值**：中

iOS / Android App。同套程式碼一份維護。可加：
- 生物辨識登入
- 相機 OCR 學生名單
- 推播
- App Store / Google Play 上架（教師更信任）

#### J3. 離線評語暫存 ⭐⭐⭐
**預估**：4 小時 ｜**價值**：高

PWA 已支援基本離線，但「沒網時寫的評語」沒暫存。加 IndexedDB queue：
- 離線時把 generate request 丟進 queue
- 上線時 service worker 自動 replay
- 失敗的有 toast 提示

#### J4. 安裝引導優化 ⭐
**預估**：2 小時

iOS Safari 沒有 add-to-home-screen 自動 prompt，可加教學截圖（一次顯示，dismiss 後不再出現）。

## 📐 設計原則總結（從 v2.10.0 + v2.11.0 學到的）

1. **基建優先於功能**：先把通知 / 監控 / 錯誤告警基建立好，後續所有功能都能直接套用
2. **best-effort 不影響主流程**：通知失敗絕不能讓使用者主流程掛掉
3. **節流是必須的**：60 秒 throttle 防洗版，未來所有外部通知都該套
4. **UX 要實機驗證**：build 過 / curl 200 不代表卡片好看，每次 UI 改動都要請使用者實機看一眼
5. **Skill 是長期資產**：踩雷經驗寫進 skill 庫，未來新專案自動避雷
6. **Secret 永遠走 Manager**：絕不寫 `.env` / `git` / `chat`，永遠 pipe 到 Secret Manager
7. **公開設計也是安全策略**：Firebase Web Key 公開是設計使然，真正的保護來自 referrer + API restrictions 不是保密
8. **設計系統先抽 token，再做 atoms，最後重寫頁面** ← v2.11.0 12 batch 順序的關鍵
9. **0 破壞性 props 改動是設計系統升級的底線**：商業邏輯絕不動，視覺重寫不影響資料層
10. **`HANDOFF.md` 是設計師與工程師的合約**：所有 token / 變體 / RWD 斷點寫下來，避免口頭傳達失真
